# Golden Project 架构基线

## 1. 架构目标

这套架构不是为了快速拼功能，而是为了支撑企业线下人员管理软件的长期稳定迭代。

架构必须同时满足：

- 主数据稳定
- 多入口协同
- 规则集中
- 板块可调整
- 渠道可扩展
- 回滚可执行

## 2. 技术栈

- 后端：NestJS
- 数据库：PostgreSQL + Prisma
- Web 控制台：静态页面 + 后端路由托管，后续可平滑升级
- 小程序：独立前端入口，调用统一后端 API
- 外部集成：飞书 / 微信 / Agent
- 部署：Linux 服务器 + PM2 + Nginx

## 3. 架构分层

### 3.1 平台基础层

职责：

- 配置读取
- 数据库连接
- 统一异常处理
- 鉴权与权限
- 审计与日志
- 定时任务与后台任务

建议目录：

- `src/platform/config`
- `src/platform/prisma`
- `src/platform/auth`
- `src/platform/audit`
- `src/platform/http`
- `src/platform/jobs`

### 3.2 领域层

职责：

- 定义稳定业务对象
- 实现核心业务规则
- 对外暴露稳定用例

领域对象至少包括：

- Project
- ProjectMember
- ProjectModule
- Task
- TaskUpdate / TaskLog
- Notification
- Event
- Risk
- RuntimeState

注意：

- 领域层是系统事实源
- UI、小程序、Agent 都只能消费或调用领域能力

### 3.3 数据访问层

职责：

- 只做数据库读写
- 不承载业务规则
- 不直接面向控制器

建议：

- 每个核心领域都逐步补 repository / dao
- 至少把复杂查询从超大 service 中拆出去

### 3.4 接入层

职责：

- HTTP Controller
- 小程序入口
- 飞书 / 微信 webhook
- Agent workflow 接口

要求：

- 只做输入接入、参数校验、调用领域服务、返回响应
- 不直接拼复杂业务

### 3.5 渠道适配层

职责：

- 飞书适配
- 微信适配
- 小程序适配
- Agent 适配

原则：

- 渠道适配层不能重新定义任务、人、项目规则
- 渠道适配层只做“格式转换 + 调用领域服务”

## 4. 当前建议模块边界

### 核心领域模块

- `projects`
  - 负责项目生命周期
  - 不负责文件系统细节、不负责 dashboard 拼装、不负责成员全部逻辑

- `project-members`
  - 负责成员加入、身份、角色、临时人员、认领关系

- `project-modules`
  - 负责模块树、模块排序、负责人关系

- `tasks`
  - 负责任务创建、分配、确认、进度更新、完成、取消、日志

- `notifications`
  - 负责通知入库、读取状态、分发记录

- `events`
  - 负责事件流入、识别、审核、确认

- `risks`
  - 负责风险识别、状态流转、风险视图

- `runtime-state`
  - 负责页面运行态、结构树运行态、身份 claim 等非主数据状态

### 支撑模块

- `dashboard`
  - 只做聚合视图，不定义业务真相

- `mini-app`
  - 只做小程序入口编排，不直接持有主规则

- `integrations/feishu`
- `integrations/wechat`
- `integrations/agents`
  - 只做渠道接入和适配

## 5. 数据源优先级

必须明确区分：

### 5.1 主数据

主数据是系统真相，包含：

- 项目
- 成员
- 模块
- 任务
- 任务日志
- 通知记录
- 风险
- 事件

这些数据必须通过明确领域接口维护。

### 5.2 运行态数据

运行态数据包括：

- 结构树展示状态
- 身份认领状态
- intake 快照
- 页面临时展示辅助数据

运行态数据可以调整，但不能替代主数据语义。

### 5.3 外部输入数据

外部输入数据包括：

- 飞书消息
- 微信消息
- 小程序操作
- Agent 建议

外部输入不是系统真相，只是候选输入，必须经过领域规则吸收。

## 6. 核心数据流

标准数据流应为：

`外部输入 / UI 输入 -> Controller / Adapter -> 领域服务 -> Repository / Prisma -> 主数据`

展示流应为：

`主数据 + 运行态数据 -> 聚合服务 -> 控制台 / 小程序 / Agent`

禁止：

- 前端页面直接定义主数据结构
- 渠道适配层跳过领域服务直接写主数据
- 用运行态 JSON 长期替代正式领域模型

## 7. 配置管理规则

必须引入统一配置模块。

要求：

- 所有环境变量集中读取
- 每项配置有明确用途
- 禁止在业务 service 零散 `process.env`
- 区分开发 / 测试 / 生产配置
- 删除密码、第三方密钥、回调 token 不能有业务默认值

## 8. 错误处理规则

必须补齐：

- 统一异常过滤
- 统一错误响应结构
- 参数错误 / 权限错误 / 资源不存在 / 集成失败 / 系统异常的统一口径

目标：

- 前端、小程序、Agent 能依靠统一错误语义处理失败
- 排障时能快速定位失败层级

## 9. 权限模型

至少分清：

- 系统管理员
- 项目管理员
- 模块负责人
- 执行人员
- 临时人员
- 外部接入身份

必须明确哪些操作需要什么权限：

- 删除项目
- 修改项目结构
- 分配任务
- 修改任务负责人
- 读取项目文件
- 修改运行态
- 触发外部消息与 Agent 工作流

## 10. 日志与审计

企业软件必须保留关键行为审计。

至少记录：

- 项目创建 / 删除
- 成员加入 / 角色变更
- 任务创建 / 分配 / 状态变更
- 风险状态变更
- 外部消息吸收结果
- Agent 自动动作

原则：

- 审计日志和业务日志分开
- 关键行为必须可追溯到人或系统身份

## 11. 部署与回滚

必须明确：

- 服务启动方式
- PM2 进程名
- Nginx 入口
- 环境变量来源
- 日志查看方式
- 当前稳定提交
- 回滚步骤

要求：

- 线上版本必须能对应到 git 提交
- 不允许长期依赖脏工作区运行

## 12. 面向未来板块调整的约束

后续允许增加：

- 时间线视图
- 日历视图
- 看板视图
- 更复杂的小程序入口
- 更多 Agent 协同板块

但这些变化必须建立在下面的稳定契约上：

- Project / Member / Module / Task 的语义不变
- 状态流转规则不随页面变化而漂移
- 新板块只能消费统一 API，不自行复制一套主逻辑
- 聚合视图可以重做，底层事实源不能随意重做
