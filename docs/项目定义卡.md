# Golden Project 项目定义卡

## 1. 项目定位

Golden Project 是一套给企业线下项目使用的人员管理与执行协同软件。

它服务的不是单次 demo，而是真实项目中的人员组织、任务下发、执行跟踪、风险反馈、信息同步与现场协同。

这个软件的第一目标不是“功能多”，而是：

- 关键数据稳定
- 人员与任务关系清晰
- 多入口协同一致
- 项目执行过程中可追踪、可回看、可纠错

## 2. 主要用户

- 项目经理
- 模块负责人
- 执行人员
- 临时人员 / 兼职人员
- 管理后台运营人员
- 通过飞书、微信、小程序、Agent 接入系统的协同方

## 3. 软件要解决的问题

当前企业线下项目管理里，最容易失控的是：

- 人员身份不统一
- 临时人员接入混乱
- 任务分配与确认链路不闭环
- 多渠道消息无法形成统一任务事实
- 现场变更频繁，但系统无法稳定吸收变化
- 小程序、后台、Agent、消息入口看到的数据不一致

Golden Project 要解决的核心问题，是把“项目、人员、任务、执行状态、风险、通知”收口成统一事实源，再把这些事实安全地分发到不同入口。

## 4. 当前阶段最核心的 7 个功能

1. 项目创建与项目基础资料管理
2. 项目成员与身份管理
3. 项目模块 / 组织结构管理
4. 任务创建、分配、确认、进度更新、完成闭环
5. 风险 / 事件 / 运营信号汇总
6. 多渠道入口协同
7. 小程序与控制台的执行视图

说明：

- 所有后续功能都必须能回到这 7 个核心能力之一
- 新板块不是独立产品，必须落在既有领域能力上

## 5. 明确不做什么

当前阶段不做：

- 通用 OA 平台
- 通用 CRM
- 通用 IM 工具
- 复杂审批流平台
- 财务 / 支付 / 报销核心系统
- 无边界的“什么入口都能直接改主数据”

如果某个需求需要引入新大类能力，必须先补项目定义，而不是直接写代码。

## 6. 前后端职责划分

后端负责：

- 领域数据模型
- 核心业务规则
- 多入口统一事实源
- 权限、审计、日志、状态流转
- 外部集成适配

前端 / 小程序 / Agent 入口负责：

- 展示视图
- 输入采集
- 对后端能力的受控调用
- 渠道内体验优化

前端、Agent、小程序都不是主数据规则的定义方。

## 7. 涉及的关键能力

本项目明确涉及：

- 登录 / 身份识别
- 角色与权限
- 文件与项目资料
- 消息通知
- 任务流
- 风险与事件
- AI 辅助分析
- 飞书 / 微信 / 小程序 / Agent 外部接口

## 8. 稳定性优先级

这是企业用的线下人员管理软件，稳定性优先级高于开发速度。

必须遵守：

- 主数据优先稳定，展示层允许逐步调整
- 先保证任务链路可靠，再扩展视觉和交互
- 先保证结构清晰，再增加智能化与自动化
- 任何新入口都不能绕过统一领域规则直接改数据库语义

## 9. 允许后续调整的范围

允许调整：

- 控制台页面布局
- 小程序页面结构
- Agent 交互方式
- 模块展示方式
- 时间线 / 日历 / 看板等视图形态
- 外部集成适配器实现

不允许随意调整：

- 项目、成员、模块、任务、通知、风险这些主领域对象的语义
- 任务状态机
- 人员身份归属规则
- 多入口写主数据的边界
- 审计与追踪要求

## 10. 本项目的产品约束

Golden Project 不是“页面集合”，而是“线下项目执行的统一数据底座 + 多入口协同系统”。

因此后续所有开发必须满足：

- 板块可以调整
- 交互可以升级
- 接入可以扩展
- 但核心领域模型、状态规则、权限边界必须越来越稳定，而不是越来越松
