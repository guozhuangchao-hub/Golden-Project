# Golden Project 框架验收清单

这份清单用于判断 Golden Project 是否允许继续进入业务开发。

验收口径：

- `通过`：已满足要求，可作为稳定基线的一部分
- `待改`：方向对，但还未收口
- `未实现`：当前没有建立

当前结论基于仓库现状，不是理想状态判断。

## 1. 项目定义

### 1.1 软件定位是否清楚

- 结论：`通过`
- 证据：已补 `docs/项目定义卡.md`

### 1.2 主要用户、核心问题、核心功能是否清楚

- 结论：`通过`
- 证据：已补 `docs/项目定义卡.md`

### 1.3 不做什么是否明确

- 结论：`通过`
- 证据：已补 `docs/项目定义卡.md`

## 2. 架构基线

### 2.1 技术栈、部署方式是否明确

- 结论：`待改`
- 证据：已有 `docs/backend-mvp-design.md`、`docs/current-project-state.md`，但仍需补正式部署 / 回滚 / 环境变量文档并更新到当前状态

### 2.2 模块划分是否清楚

- 结论：`待改`
- 证据：已补 `docs/架构基线.md`，但现有代码尚未按目标边界拆分

### 2.3 数据流是否清楚

- 结论：`待改`
- 证据：文档已定义主数据、运行态、外部输入三层；代码里仍存在运行态和主数据边界混用风险

### 2.4 配置、错误处理、权限模型是否成体系

- 结论：`未实现`
- 证据：
  - 尚无统一配置模块
  - 尚无统一异常过滤器
  - 尚无清晰守卫 / 权限模型落地

## 3. 开发规则

### 3.1 代码放置规则是否明确

- 结论：`通过`
- 证据：已补 `docs/开发规则.md`

### 3.2 主数据与运行态边界是否明确

- 结论：`通过`
- 证据：已补 `docs/架构基线.md`、`docs/开发规则.md`

### 3.3 接口、配置、集成、变更规则是否明确

- 结论：`通过`
- 证据：已补 `docs/开发规则.md`

## 4. 目录责任

### 4.1 controller 是否只负责接入

- 结论：`待改`
- 证据：大部分 controller 还算薄，但仍存在 `any` 入参和下载 / 打开文件等耦合接口未完成约束

### 4.2 service 是否职责清晰

- 结论：`未实现`
- 证据：
  - `projects.service.ts` 体量过大，职责混合
  - 多个 service 同时承担聚合、规则、数据访问、适配责任

### 4.3 repository / dao 是否建立

- 结论：`未实现`
- 证据：当前主要直接由 service 操作 Prisma

## 5. 分层边界

### 5.1 外部接入是否通过统一领域规则入库

- 结论：`待改`
- 证据：已有模块划分，但仍缺正式的领域边界与统一入口约束

### 5.2 页面 / 小程序 / Agent 是否只消费统一事实源

- 结论：`待改`
- 证据：方向上成立，但当前运行态 JSON 和聚合逻辑仍容易成为事实源替代品

## 6. 接口与数据规范

### 6.1 DTO 校验是否覆盖核心写接口

- 结论：`待改`
- 证据：已有不少 DTO，但部分高风险接口仍使用 `any`

### 6.2 返回结构与错误结构是否统一

- 结论：`未实现`
- 证据：当前没有统一响应 / 错误结构约束

### 6.3 主数据命名与状态机是否稳定

- 结论：`待改`
- 证据：Prisma schema 已形成一定基础，但仍需正式固化核心领域语义

## 7. 权限与安全

### 7.1 高风险操作是否有明确权限控制

- 结论：`未实现`
- 证据：当前未形成完整 guard / policy 体系

### 7.2 敏感配置是否集中管理

- 结论：`未实现`
- 证据：多个 service 中直接读取环境变量

### 7.3 敏感操作是否可审计

- 结论：`待改`
- 证据：部分任务与事件链路有日志意识，但系统级审计规范未建立

## 8. 可运维性

### 8.1 服务启动、入口、日志、回滚方式是否明确

- 结论：`待改`
- 证据：共享大脑中已有服务器信息，但仓库内正式文档不足

### 8.2 线上版本是否能稳定映射到 git 提交

- 结论：`待改`
- 证据：本轮已把服务器同步到正式提交，但流程还未制度化

### 8.3 本地备份与线上状态是否可核对

- 结论：`待改`
- 证据：本轮已验证一次，但仍缺正式流程文档

## 9. 测试与基线

### 9.1 是否有最小自动化测试基线

- 结论：`未实现`
- 证据：`npm test` 目前无测试用例

### 9.2 是否有稳定基线版本可供后续开发

- 结论：`待改`
- 证据：已有 git 提交基线，但文档、配置、权限、测试尚未同步收口

## 10. 总结结论

### 是否允许继续直接推进新业务功能

- 结论：`不允许`

### 原因

- 当前文档基线刚补齐，但代码基线尚未收口
- 模块职责、配置、权限、错误处理、测试基线都未完全建立
- 如果继续直接堆功能，会继续扩大返工范围

### 允许优先推进的事项

1. 收口项目正式基线文档
2. 拆分 `projects.service.ts`
3. 建立统一配置与权限入口
4. 固化主数据 / 运行态边界
5. 建立最小冒烟测试集

### 达到什么状态后可恢复业务开发

至少满足：

- 配置集中化
- 高风险接口去掉 `any`
- `projects` 大模块完成第一轮拆分
- 核心链路有最小回归测试
- 部署 / 回滚 / 环境变量文档成型
