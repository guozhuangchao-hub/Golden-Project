# Golden Project 框架下一步动作

更新时间：2026-06-12

这份清单承接“框架验收清单”的当前结论，记录在框架已达到当前可持续基线后，仍值得继续优化但不再阻塞主线业务开发的事项。

## 1. 长期优化：继续压低 service / repository 耦合

目标：在不影响业务推进的前提下，继续改善代码整洁度。

优先处理：

1. 继续把复杂查询往 repository / 专用 service 下沉
2. 继续压小仍偏厚的门面 service
3. 持续保持业务链路弱类型为零

完成标准：

- 新业务迭代不会重新把 service 写回“大杂烩”形态
- 数据访问层继续保持可扩展

## 2. 长期优化：扩大权限与审计可视化

目标：让后续开发更容易判断入口是否满足权限与审计要求。

建议动作：

1. 维护高风险入口清单
2. 记录 guard 与 audit 覆盖面
3. 新增高风险入口时同步更新清单

完成标准：

- 后续功能开发不会绕开既有权限链

## 3. 长期优化：补更广的自动化回归

目标：把当前框架基线的稳定性转化成更高的自动化保障。

建议顺序：

1. 权限拒绝 / 放行回归
2. 运行态读写回归
3. 飞书 / Agent / 小程序关键链路回归

完成标准：

- 后续框架调整更容易快速验证

## 4. 长期优化：维持运维工作区规则

目标：保持线上环境长期可回放、可核对、可解释。

当前规则：

- `package-lock.json` 应与仓库版本一致
- `_server_snapshots/` 允许保留，但只作为归档目录存在

建议动作：

1. 每次部署后复核工作区是否只剩允许残留
2. 新增归档时遵守统一命名和说明规则

完成标准：

- 服务器工作区状态长期可解释

## 5. 当前结论

当前 GP 框架状态已达到约 `95 / 100` 的可持续基线。

现在更适合：

- 稳定推进业务功能
- 做跨端联动功能
- 做更大范围的交互和视图改造

不建议回退去做的事：

- 再起一轮大范围结构性重构
- 放弃现有权限 / 审计 / 配置收口规则
- 在未说明理由的情况下重新引入线上脏工作区依赖
